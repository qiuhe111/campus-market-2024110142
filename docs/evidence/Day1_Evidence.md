## 学习收获

1. **Vue 3 项目启动流程**
   以前写 Vue 只是照搬模板，今天读 main.ts 才真正理解 `createApp` → `app.use()` → `app.mount('#app')` 每一行在做什么。Pinia 和 Router 都是通过插件机制注册，这个"插件体系"是 Vue 3 扩展能力的基础。

2. **SPA 路由工作原理**
   Router 的作用不只是配置路径，它通过 `<RouterView>` 动态渲染当前匹配的组件，实现无刷新页面切换。路由守卫则可以在导航前后执行逻辑（如登录校验）。

3. **目录结构设计的工程意义**
   按职责分层（views/components/stores/api）不是形式主义——当组件、状态、API 请求混在一起时项目会迅速变得难以维护。这种分层让每段代码都有"固定归属"，新成员也能快速定位。

4. **AI 协作的节奏**
   通过 Opencode 逐步分析目录、代码、技术栈，发现好的 Prompt 应该是"分步骤、明确上下文"的，而不是一次性扔一个大问题。

## 实验思考

### 1. 为什么课程没有要求第一天立即开发页面？

第一天没有直接写页面，而是安排理解背景、分析目录、阅读核心代码、做项目规划，这本身就是软件工程"**先想清楚再动手**"的体现。如果第一天就写页面，可能出现的问题：
- 不清楚项目整体结构，代码放错目录
- 不理解路由机制，页面写好了却无法正确导航
- 没有规划依赖顺序，写到后面发现前面缺东西要返工

课程设计上，Day1 更像"**施工蓝图阶段**"——先把项目全貌看清、把目录用途搞懂、把开发顺序理清，后续 6 天的编码才有方向感。这也符合真实项目中**需求分析 → 架构设计 → 编码实现**的流程。

### 2. 软件工程中的"理解项目"为什么比"开始编码"更重要？

类比盖房子：
- **理解项目** = 看图纸、勘测地基、规划施工顺序
- **开始编码** = 直接砌墙

没有理解就编码，相当于没有图纸就开始施工——可能墙砌歪了、材料用错了、甚至建在了错误的位置上。具体到软件工程：
- **理解业务场景**：知道系统要解决什么问题，才能写出对的功能
- **理解技术架构**：知道为什么用 Pinia、Router 怎么工作，才能写对的代码
- **理解目录约定**：知道每个文件该放哪，团队协作时别人才找得到
- **理解依赖顺序**：路由 → 用户模块 → 商品模块，顺序错了就卡住

**"理解"节省的时间远远多于"直接编码"多花的时间。** 今天花半天理解项目，可能为后面 6 天省下反复重构的时间。

### 3. AI 在软件开发中应该承担什么角色？

今天的协作体验下来，我认为 AI 的角色应该是 **"高级辅助"而非"自动驾驶"**：

| AI 擅长的 | 人应该做的 |
|-----------|-----------|
| 快速检索信息（技术栈、API） | 决定"做什么"（需求判断、方案选型） |
| 生成结构化输出（目录分析、代码模板） | 验证 AI 输出的正确性（版本差异、项目特定配置） |
| 解释代码逻辑 | 理解背后的设计原理 |
| 生成重复性代码（CRUD 模板） | 编写核心业务逻辑 |

今天的实践中，AI 帮我快速分析了项目结构和技术栈，但在判断"为什么这样设计"时仍然需要我自己思考。**AI 是"效率工具"，不是"替代思考的工具"。** 用好 AI 的关键是：知道自己要什么，能判断 AI 给的答案是否靠谱。

### 4. 今天最大的收获是什么？

最大的收获是 **"先理解再动手"的工程思维**。

之前做项目总是急着写代码，经常写到一半发现结构不对要重构。今天按照课程的节奏走下来，虽然没有写一行业务代码，但对项目的整体认知非常清晰：
- 知道 src/api/components/router/stores/views 各管什么

- 知道 main.ts 的启动流程
- 知道 RouterView 在哪里渲染页面
- 知道接下来 6 天每天该做什么

这种"全局掌控感"是以前直接上手编码时从来没有过的。另一个收获是**AI 协作的边界感**——AI 能快速给答案，但验证和决策还是要靠自己。
